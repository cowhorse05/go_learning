# CDN 笔记：反向代理、Redis、整体架构

## 一、没有 CDN 的时候，用户怎么访问网站？

```
用户(北京) ──── 很远 ────> 源站(深圳)
```

问题很明显：距离越远，网页加载越慢。而且所有用户都打同一个服务器，服务器扛不住。

---

## 二、CDN 做了什么？

CDN（内容分发网络）的核心思想一句话：**把内容提前存到离用户最近的节点上**。

```
用户(北京) ──> 北京CDN节点(有缓存，直接返回) 
用户(上海) ──> 上海CDN节点(有缓存，直接返回)
用户(广州) ──> 广州CDN节点(没缓存，回源站拉一下)
                                        │
                                      源站(深圳)
```

效果：大部分请求在边缘节点就处理掉了，源站压力小，用户访问快。

## 四、反向代理是什么？

先说**正向代理**：你主动挂代理访问外部网站，你知道代理的存在。

```
你 ──> 代理服务器 ──> 目标网站
(你知道代理)      (网站不知道你)
```

**反向代理**反过来：用户不知道代理的存在，用户以为自己访问的就是目标服务器。

```
用户 ──> 反向代理(看起来像最终服务器) ──> 后端真实服务器
(用户不知道后面还有东西)                (服务器不知道用户)
```

### 反向代理在 CDN 里做什么？

| 职责 | 解释 |
|------|------|
| **统一入口** | 用户只看到一个 IP，实际上后面可能有几百台机器 |
| **负载均衡** | 把请求分给多台后端，哪台空闲分给哪台 |
| **SSL 卸载** | HTTPS 加解密在代理层完成，后端只处理 HTTP，省算力 |
| **缓存** | 静态文件（图片/CSS/JS）直接返回，不麻烦后端 |
| **安全防护** | 隐藏后端真实 IP，挡住恶意请求 |
| **压缩** | 把响应内容 gzip 压缩后发给用户，省流量 |

### 一句话总结

> 反向代理 = 后端的"前台"，用户跟你打交道，你去找后面的人干活。

## 五、Redis 在 CDN 里干什么？

Redis 是一个**内存数据库**，读写极快（微秒级）。

### 为什么不能只靠后端服务器自己算？

后端每次处理请求都要做很多事：查数据库、拼接页面、校验权限… 每次都来一遍太慢了。

### Redis 的作用

```
用户请求: /api/hot-news
         │
         ▼
    先问 Redis: "hot-news 有缓存吗？"
         │
    ┌────┴────┐
    ▼         ▼
  有缓存     没缓存
  直接返回     │
              ▼
          后端计算 "hot-news"
              │
              ▼
          把结果存进 Redis（下次就有了）
              │
              ▼
          返回给用户
```

### Redis 具体缓存什么？

| 数据类型 | 例子 | 为什么放 Redis |
|----------|------|----------------|
| 热点数据 | 首页推荐列表 | 每个人都看，重复算浪费 |
| Session | 用户登录状态 | 每次请求都要验证，必须快 |
| 计数器 | 文章阅读量 | 每读一次 +1，写数据库太慢 |
| 限流 | 用户1秒最多请求10次 | Redis 天然适合计数 |
| 临时数据 | 验证码 | 5分钟过期的数据 |

### 为什么不把所有东西都放 Redis？

Redis 贵在**内存**，内存比硬盘贵很多。所以只放"热数据"（经常被访问的），冷数据（很少被访问的）还是放数据库或硬盘。

### 一句话总结

> Redis = 全局快捷方式，把算好的结果存起来，下次直接用，不用重新算。

## 六、一个请求的完整生命周期

```
1. 用户在浏览器输入 www.example.com
2. DNS 解析，返回最近的 CDN 节点 IP
3. 请求到达 CDN 节点的反向代理(Nginx)
4. 反向代理判断：
   ├── 静态文件(图片/CSS/JS) → 从本地缓存直接返回（最快）
   └── 动态请求(API) →
       ├── 先问 Redis 有没有缓存结果
       │   ├── 有 → 直接返回（很快）
       │   └── 没有 → 去源站后端拿结果
       │       ├── 后端查数据库计算
       │       ├── 结果写入 Redis（下次就快了）
       │       └── 返回给用户
       └── 返回
```

可以看到，**Redis 快于后端计算**，**本地静态缓存最快**。CDN 的分层缓存就是这个意思——能用近的就不用远的。

---

## 七、核心概念三句话

| 概念 | 大白话 |
|------|--------|
| **CDN** | 在各地放缓存服务器，用户就近访问，快 |
| **反向代理** | 后端的"前台接待"，用户找你，你分活给后面的人 |
| **Redis** | 内存里的"草稿纸"，算过的东西记下来，下次直接用 |

## 八、和你的 day1 项目的关系

你写的 HTTP 服务有三个接口：

```
/        → homeHandler
/health  → healthHandler
/info    → infoHandler
```

如果把它部署到 K8s 上，架构大概是这样：

```
用户 → CDN/反向代理(Nginx) → K8s Service → 你的 Pod(Go程序)
                                  │
                              Redis(缓存热点查询)
```

现在你的程序很简单，还用不到 Redis。但理解整个链路对你后续学习很有帮助——后面你会学 Ingress（K8s 里的反向代理）、学怎么配合 Redis、学怎么打镜像部署。
